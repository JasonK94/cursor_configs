# Cursor 협업 워크플로

이 가이드는 `cinit` 실행 후 Cursor 기반 개발을 어떻게 시작하고 관리할지 설명합니다. 영어 버전은 `docs/cursor_workflow.md`를 참고하세요.

## 1. 빠른 시작 (약 5분)

1. **생성된 파일 확인**
    - `context.md` (전역 규범)
    - `projects/<project_name>/context.md` 등 프로젝트 전용 컨텍스트(존재할 경우)
    - `DEVLOG.md`, `CHANGELOG.md`, `tasks/` 하위 브리프
    - `cinit` 실행 시 선택한 언어에 따라 생성된 `README.md`, `README_Korean.md` 등
2. **초기 프롬프트 준비**
    - `context.md` → *Context Hierarchy & Roles*에 명시된 순서대로 파일을 드래그&드롭하여 Cursor에 제공
    - 현재 TODO 또는 작업 브리프 포함
3. **플랜 확인**
    - 에이전트에게 이해한 내용과 제안 계획을 요약하도록 요청
    - 세션 시작 사실을 `DEVLOG.md`에 기록

## 2. 상세 매뉴얼

### 2.1 부트스트랩

1. `cinit`(또는 프로젝트 전용 초기화 스크립트) 실행
2. 질문에 따라 문서 언어 선택: `1 = 한·영`, `2 = 영어`, `3 = 한국어`
3. 추가 템플릿이 필요하면 `templates/`에서 복사
4. 프로젝트 이름, 범위, 마일스톤 등 메타데이터를 채운 뒤 다른 에이전트에게 전달

### 2.2 에이전트에게 컨텍스트 제공

1. 피드 순서를 따릅니다: 루트 컨텍스트 → 메타 에이전트 컨텍스트 → 프로젝트 컨텍스트 → 로그 → 작업 브리프
2. 도메인 문서(디자인, API 등)는 필수 컨텍스트 이후에 추가
3. 공유 전에 각 파일이 최신 상태인지 확인하고 필요하면 갱신

### 2.3 작업 수행

1. 에이전트는 (사소한 요청이 아니라면) TODO 계획을 작성하고 검토 요청
2. 작은 단위로 구현하며 진행 상황을 자주 요약
3. 주요 작업 후에는 다음 문서를 업데이트
    - `DEVLOG.md`: 의사결정, 진행 상황, 다음 단계
    - `project_context.md`: 새로운 상태나 제약 사항
    - `CHANGELOG.md`: 사용자 영향이 있는 변경 사항

### 2.4 핸드오프 & 마무리

1. 세션 종료 전, 관련 작업 브리프에 상태 업데이트
2. 미완료 작업이 있으면 남은 단계와 리스크를 명시
3. 사람 협업자는 변경 사항을 검토·테스트 후 저장소 정책에 따라 병합

## 3. 역할과 책임

- **메타 에이전트**: `context.md`, `DEVLOG.md`, 템플릿 등 저장소 전반 규범을 관리
- **프로젝트 에이전트**: 특정 프로젝트 범위 내 작업 수행, 프로젝트 컨텍스트 최신화
- **인간 관리자**: 결과 검증, 병합 승인, 운영 품질 확보

## 4. 협업 팁

- 문서는 간결하게 유지하고, 중복 대신 링크를 활용
- 브랜치 네이밍을 통일 (`proj/<project>`, `chore/<project>/<agent>/<topic>`)
- 반복되는 검사(린트, 포맷, 템플릿 검증)는 자동화 고려
- `cupdate`: 로컬 `cursor_configs` 프레임워크(`~/.cursor_configs`)를 최신 상태로 유지하므로 주요 작업 전에 실행하는 것이 좋습니다.

워크플로가 발전하면 이 문서를 업데이트하고, 주요 변경 사항은 `DEVLOG.md`에 기록하세요.
